Technical Design Document

Introduction

Purpose

This platform serves as a centralized hub for students seeking tech CO-OPs and internships, matching job opportunities tailored to their resumes. InternSync aims to simplify the application process by aggregating tech-related internship and CO-OP positions from LinkedIn across Canada and the USA.

What I Plan to Achieve

Building InternSync is an opportunity for me to experience the full software development lifecycle firsthand. I intend to familiarize myself with the intricacies of building a full-stack application, covering the stages of planning, designing, and implementation independently.

Scope

  1. Job Aggregation
    • Utilize the ScrappingDog API, specifically its custom LinkedIn request feature, to aggregate internship job listings.
  1. Resume Parsing and Matching
    • Design and implement an algorithm to parse user resumes.
    • Match parsed resumes with the aggregated job listings to recommend the most suitable internship opportunities.
  1. Technology Stack
    • Frontend: Angular, Bootstrap
    • Backend: Node.js, Express, and TypeScript
    • Database: MongoDB, Amazon S3
  1. User Functionalities
    • Account Creation: Allow users to create, edit, and manage their profiles.
    • Resume Upload: Provide an interface for users to upload and store their resumes.
    • Job Match Algorithm: Based on the parsed resume data, the algorithm will suggest relevant job listings to the user.
    • Application Tracker: A feature for users to monitor the status and progress of their job applications.
    • Search: Index jobs based on phrases and keywords.
    • Sort By: Filter jobs based off resume, location, date posted and company.

Boundaries

Job Type

  • The platform will exclusively focus on internships and will not cater to full-time or other job types.

Job Sources

  • Only job listings from LinkedIn will be aggregated. There will be no integration with other job platforms or sources.

Geographic Restriction

  • Jobs will be mostly catered to students seeking an internship in Canada or the United States.

Application Process

  • The platform will provide listings and matching but will not handle or support the direct application process to the jobs. Users will be redirected or provided with information to apply on the primary platform (e.g., LinkedIn).

Design Considerations

Assumptions

  1. User Authenticity: It's assumed that users signing up for the platform will be legitimate students seeking internships and will provide accurate details on their profiles.
  1. Resume Consistency: Users will upload resumes in a format such as .pdf or .docx, and their content is assumed to follow a standard resume structure.
  1. LinkedIn Data Integrity: The ScrappingDog API's ability to accurately scrape data from LinkedIn is assumed to be reliable and up-to-date.
  1. Regulations Compliance: The platform is operating under the assumption that the use of the ScrappingDog API and the scrapping of LinkedIn does not violate any terms of service or legal regulations.
  1. User Engagement: Users are expected to return to the platform regularly to check for updated listings or monitor their application tracker.
  1. Cloud Service Availability: Amazon S3, Heroku, and MongoDB atlas integrated with the platform are assumed to have high availability.

Constraints

  1. Technical:
    • Rate Limiting: The ScrappingDog API might have limitations on the number of requests that can be made in a specific time frame.
    • Data Storage: Given that resumes might contain sizable amounts of data and that you'll be storing job listings, there might be limitations on the data that can be stored in MongoDB and Amazon S3.
    • Data Parsing: The complexity and variety of resumes may pose challenges in designing an algorithm that effectively parses and categorizes all types of resume data.
    • Frontend Compatibility: Ensuring that the platform remains responsive and accessible across different devices and browsers can be challenging.
  1. Financial:
    • API Costs: There may be costs associated with using the ScrappingDog and TextRazor API’s, especially at higher request volumes.
  1. Operational:
    • Data Privacy: Ensuring that user data, especially resumes, remain private and secure is paramount.
    • Up-to-date Listings: Maintaining the freshness of job listings to ensure users get timely opportunities.

Architecture Design

Dashboard

Jobs

The Jobs component provides users with a streamlined view of available job listings. Through this component, users can engage with various job listings, view a summary, and load more jobs as they scroll through the list.

Pagination

Instead of overloading users with a long list, they're presented with chunks of job listings, allowing for smoother interaction and reduced initial load times.

  • MongoDB Queries: Utilize the $skip and $limit operations to fetch specific sets of job listings. As the user scrolls, adjust the parameters to fetch the next set.

Job Listing

A concise job summary is displayed in the Card Component. Upon clicking a particular job card, detailed information about the job is showcased in the Job Details Component.

Component Logic

  • State Management: Leveraging an NgXs store to maintain and manage the job-related states. The @Select decorators choose specific slices of state, which include the available jobs, the currently selected job, loading state, and total job count.
  • Pagination Management: The currentPage variable manages which set of jobs to fetch next. As the user scrolls, the page number increments, leading to the subsequent batch of job listings being loaded.
  • Lifecycle Hook & Initialization: Upon component initialization (ngOnInit), the component dispatches an action to load the initial set of jobs and fetch the total count of jobs available.
  • Job Interaction: The selectJob method is triggered when a user clicks on a job card, updating the current selected job state.

UI/UX Implementation

  • Responsive Job Display: The job listings are wrapped within the jobs-container that uses the infinite-scroll directive. This directive listens for user scrolling events to trigger the loading of more jobs.
  • Loading Indicators: Strategically placed loading spinners (app-loading-spinner) give users feedback when job data is being fetched. Two scenarios are considered:
    1. When no jobs have been loaded, and the system is in the process of fetching them.
    1. When the user has scrolled down, indicating the fetching of the next batch of jobs.
  • Job Count Display: A top bar displays the current count of displayed job listings against the total number of job listings available.

Job Details

The Job Details component is a detailed display that provides an in-depth view of each job listing. It serves as the focal point for users who want a deeper understanding of a particular job position, on click.

Reactive Data Management: The component uses Observables for asynchronous data operations which provides seamless data integration with the component template, ensuring real-time UI updates.

Separation of Concerns: The component ensures SoC by utilizing selectors for fetching data, keeping the component logic clean and the state management decoupled.

Dynamic Data Presentation: Through string interpolation, the template adapts and presents data dynamically.

Utility Functions: The component houses utility functions like formatDescription to enhance the presentation of the job description, making it more readable.

Detailed Breakdown

Component Initialization: On the component's initialization ngOnInit, it prepares itself to display the job details for the first job in list.

State Management & Data Flow:

  • Selectors: Selectors such as this.store.select(state => state.jobs.selectedJob) to extract the currently selected job details from the store.
  • Reactive Data Binding: selectedJob$ and isLoading$, ensures a link between the data layer and the UI, allowing for the currently selected job to be displayed and a loader while fetching the data.

User Interactions: The openJobLink function offers a way for users to visit the actual job posting.

Performance: By making use of selectors and Observables, the component ensures optimal performance. The Observables, in particular, ensure that the UI updates only when necessary, preventing unnecessary re-renders and promoting a smoother user experience.

Quick Apply

The QuickApply component functions provides insights into individual job listings, such as "People Also Viewed" and "Similar Jobs" categories.

Feature Breakdown

Mode Determination

The component has two primary display modes:

  • Similar Jobs: Job positions that are similar in nature to the currently selected job.
  • People Also Viewed: Job listings that were also checked out by users who viewed the current job.

Job Interaction

Each job listing, whether from the 'similar jobs' or 'people also viewed' category, comes with a quick apply feature, streamlining the user's application process.

Loading State Management

To enhance UX, a loading spinner is integrated, indicating when job data is being fetched.

Component Logic

  • State Management: The component uses the NgXS store to manage and maintain job-related states.
  • Component Mode: The @Input decorator for the mode variable allows 2 different UI displays.
  • Lifecycle Hook & Initialization: During component initialization ngOnInit there's a subscription to the selectedJob$ Observable. This ensures that once a job is selected, the isInitialLoad flag is toggled, controlling the display logic in the template.

UI/UX Implementation

  • Mode-Based Rendering: ngIf is used to conditionally render job listings based on the current mode.
  • Job Card Integration: The app-job-card directive is utilized for rendering job listings. With input bindings like [job], [mode], and [selectedJob], this facilates adjusting the display and behavior based on the input data it receives.

Utilities Bar

The Utilities Bar component offers a suite of navigation and filtration tools that tailor the job browsing process according to the user's specific needs.

Feature Breakdown

Search Bar: A responsive search bar allows users to delve into the vast listings by targeting specific keywords, titles, or descriptions.

Sort By: Customize the display order of job listings based on their preferences, be it by date, relevance, or other criteria.

Application Tracking: Monitor their application's progress, from the initial submission to the final decision, keeping the user in the loop at all times.

Component Logic

  • Database Interaction
    • Full-Text Search: Leveraging MongoDB's full-text search capabilities.
    • Sort Mechanism: Utilizing MongoDB's $sort aggregation pipeline stage offers versatile sorting options.

Dashboard Header

User Menu: A simple dropdown. On clicking 'Logout', clear the user session. For other functionalities like settings, redirect to the respective page.

Resume Upload: Allow users to upload their resume. Stored in Amazon S3, and keep a reference in MongoDB to created signed urls.

Resume Matching

  • Parsing Resumes: Using pdf-parse for Node.js to extract text from resumes.
  • Tokenizing Resumes: Using TextRazor API to break down the resume into tokens (words or phrases).
  • Scoring: For each job, give a score based on how many tokens match the job's requirements or description.
  • Display: Show jobs to the user based on their matching score.

High Level View of Algorithm

  • For each job, compare its required skills, qualifications, and concepts against the user's resume data.
  • Assign a score based on:
    • Exact matches of skills/qualifications.
    • Similarity matches (e.g., if the job requires "JavaScript" and the resume has "Node.js", it should be a partial match).
    • Frequency of matching terms in the resume.
  • Display jobs to the user sorted by their matching score, with the highest scores first.
  • Allow users to adjust weights or prioritize certain skills or qualifications if they wish to customize the job recommendations further.
  • Highlight matching skills and qualifications in description.

Rationale for the Data Structure

Modularity and Reusability: By having a separate structure for this, any component or function that requires only high-level information about the job can use this lightweight structure.

Extensibility: The JobDetail structure has arrays for similarJobs and peopleAlsoViewed, allowing the flexibility to add more related jobs without restructuring.

Data Flow

Job Flow Diagram

  1. Initialization:
    • JobsComponent initializes and dispatches the LoadJobs action to fetch the initial set of jobs.
    • NGXS store updates isLoading to true and calls the JobsService.
    • JobsService fetches the jobs and returns them.
    • NGXS store updates the jobs array and sets isLoading to false.
  1. Continuous Scrolling:
    • When a user scrolls to the bottom of the job list in JobsComponent, it dispatches the LoadJobs action again to fetch more jobs.
    • The process repeats (steps in Initialization) and newly fetched jobs are appended to the existing job list in the store.
  1. Job Selection:
    • A user clicks on a JobCardComponent.
    • The selected job is emitted to JobsComponent, which then dispatches the SetSelectedJob action to update the selectedJob in the NGXS store.
    • JobDetailsComponent observes changes in the selectedJob and updates its display accordingly.
  1. Loading Indication:
    • The isLoading property in the NGXS store is observed by the JobsComponent to show or hide the loading spinner.
  1. Error Handling:
    • In case of any error during the job loading process, the error is caught and logged, and isLoading is set to false.

Auth Form Flow Diagram

Handling File Uploads

  • The frontend requests a pre-signed URL from the backend.
  • Backend generates a pre-signed URL for S3 and sends it back to the frontend.
  • The frontend directly uploads the file to S3 using this URL.
  • After successful upload, frontend can send the file reference (e.g., the S3 URL) to the backend for storage in the database.

Advantages:

  • Offloads the file handling from your server, potentially saving on bandwidth and processing.
  • Faster for the user, especially if they are geographically closer to the S3 region than to your server.

3. Additional Considerations:

  • File Validation: Validation of file types client side.
  • Size Limits: Size limits for uploads.
  • File Organization: In S3 files are organized into folders, e.g., /resumes/, /profile-pics/.
  • Security: Ensure the S3 bucket is private. Files are not be publicly accessible by default. Use pre-signed URLs for temporary access when needed. Least privledge AWS credentials.

Interface Design

1. Sign In

  • Endpoint: /login
  • HTTP Method: POST
  • Description: Authenticate a user and provide them with a JWT token for further authenticated requests.

Request Payload:

{
    "email": "user@example.com",
    "password": "user_password"
}

Response: Success:

{
    "user": {
        "email": "user@example.com",
        "...": "..."
    },
    "token": "jwt_token_string"
}

Failure:

  • Status Code 401: Wrong credentials.
  • Status Code 409: Wrong credentials.

2. Sign Up

  • Endpoint: /signup
  • HTTP Method: POST
  • Description: Register a new user to the system.

Request Payload:

{
    "email": "newuser@example.com",
    "password": "user_password"
}

Response: Success:

{
    "user": {
        "email": "newuser@example.com",
        "...": "..."
    },
    "token": "jwt_token_string"
}

Failure:

  • Status Code 409: Email already exists or other signup error.

3. Fetch Jobs

  • Endpoint: /api/jobs
  • HTTP Method: GET
  • Description: Fetch a paginated list of jobs.

Query Parameters:

  • page: Page number (default is 1).

Headers:

  • Authorization: Contains the JWT token in the format Bearer jwt_token_string.

Response: Success:

{
    "success": true,
    "data": [
        {
            "jobId": "12345",
            "title": "Job Title",
            "...": "..."
        },
        "...": "..."
    ]
}

Failure:

  • Status Code 401: Unauthorized, possibly due to missing or invalid JWT token.
  • Status Code 500: Internal Server Error or other errors.

4. Total Jobs Count

The /api/jobs/count endpoint requires the user to be authenticated, i.e., the request header must include a valid JWT token.
  • Endpoint: /api/jobs/count
  • HTTP Method: GET
  • Description: Fetches the total count of available jobs. Authentication required.

Request Headers:

{
    "Authorization": "Bearer jwt_token_string"
}

Response: Success:

{
    "count": 1023
}

Failure:

  • Status Code 401: Not authenticated.
  • Status Code 500: Internal Server Error.

Third-party Integrations

  • JWT: The system uses JSON Web Tokens (JWT) for authentication. When a user logs in or signs up, they are provided with a JWT token. This token needs to be attached to the request headers for authenticated routes.

Created using Figma, allowing for experimentation with a wide range of design elements.

Security Considerations

The AuthFormComponent encapsulates the authentication and signup processes, leveraging user input and interactions to securely navigate and manage user states.

  • Dynamic Component Modes: The component can operate in either 'login' or 'signup' mode.
  • Input Validators: For both login and signup forms, validators are attached to input fields to ensure data integrity.
  • Token Storage: Upon successful authentication or signup, the user token is stored in the browser's localStorage.

Data protection and encryption methods:

Form Initialization & Validation:

  • Reactive Forms: The component uses Reactive Forms for form initialization and validation. Reactive Forms offer better customizability, unit test capabilities, and clear data flow compared to Template-driven forms.
  • Custom Validators: In the signup mode there's a custom validator MustMatch.
  • Sensitive Data Handling: When sending files (e.g., resumes, profile pictures) to the server, the component leverages Amazon S3's pre-signed URLs.
